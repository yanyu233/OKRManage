# Route C 中文示例数据与工具栏筛选设计

- 日期：2026-04-11
- 范围：Route C
- 目标：将当前英文测试数据改为中文业务示例，并为员工端、评分工作台、评分排名补回“搜索 + 年度 + 季度”工具栏

## 背景

当前 Route C 已经具备员工端、负责人端和系统管理员端的主要功能，但仍有两个明显体验缺口：

1. Prisma seed 和页面展示数据大量仍是英文示例，不利于业务演示和本地调试。
2. 员工端“我的 OKR”、负责人端“评分工作台”和“评分排名”顶部缺少时间选择与搜索工具栏，和原系统操作习惯不一致。

## 本次设计

### 1. 中文示例数据

Route C 的后端 seed 数据统一改为中文业务语义：

- 部门、科室、评价组名称改为中文
- 示例员工、负责人、系统管理员名称改为中文
- 目标、关键结果、材料名称与备注改为中文

本次只改演示数据内容，不改变数据库模型和接口字段结构。

### 2. 三页工具栏统一

在以下页面顶部补统一的工具栏结构：

- 员工端：`我的 OKR`
- 负责人端：`评分工作台`
- 负责人端：`评分排名`

统一包含：

- 搜索框
- 年度选择
- 季度选择
- 刷新按钮

布局目标：

- 工具栏位于页面标题区域上方或右侧
- 在桌面端保持一行或两行有序排布
- 在窄屏下自动换行，不压坏卡片布局

### 3. 时间选择逻辑

- 年度和季度仍沿用现有后端查询参数：`year`、`quarter`
- 切换年度或季度时，页面重新请求后端数据
- 本轮不引入新的默认季度算法，先以页面本地状态管理为主
- 初始值继续从当前页面已有常量 `2026 / 1` 起步，后续可再统一到动态默认周期

### 4. 搜索逻辑

本轮搜索统一采用“前端本地筛选当前已加载数据”的方案，不新增后端搜索接口。

员工端：

- 对当前季度返回的目标卡片做筛选
- 搜索范围：目标编码、目标名称、目标说明

评分工作台：

- 对左侧员工列表做筛选
- 对顶部目标 tabs 做筛选
- 对当前目标下关键结果卡片做筛选
- 搜索范围：员工姓名、科室、评价组、目标编码、目标名称、关键结果编码、关键结果名称、关键结果说明

评分排名：

- 对当前评价组下的排名列表做筛选
- 对右侧目标分数构成中的目标、关键结果展示做筛选
- 搜索范围：员工姓名、科室、评价组、目标编码、目标名称、关键结果编码、关键结果名称

本轮不要求跨页、跨季度、跨后端全集搜索。

### 5. 中文化范围

本轮中文化只处理 Route C 前端页面和 Route C 演示 seed 数据：

- React 页面文案
- 通用标签函数
- 页面内 placeholder、按钮、空状态、说明文案
- Prisma seed 中用于展示的演示文本

本轮不要求把后端内部状态码改中文，不要求新增国际化框架。

## 非目标

本轮不做以下事情：

- 不增加后端搜索接口
- 不改动企业微信登录流程
- 不统一改造系统管理员页面的所有中文文案编码历史问题以外的视觉结构
- 不重做季度默认值算法

## 验证标准

### 后端

- 重新 seed 后，Route C 接口返回中文业务示例数据
- 员工端、负责人端现有 e2e 测试保持通过

### 前端

- 三个页面都出现搜索、年度、季度工具栏
- 切换年度/季度会重新请求对应数据
- 输入关键词时，本地列表正确筛选
- 页面文案和主要展示字段改为中文

### 人工验证

- 系统管理员、负责人、员工三种账号都仍能正常登录
- 我的 OKR 页面可按关键词筛目标
- 评分工作台可按关键词筛员工、目标和关键结果
- 评分排名可按关键词筛员工和明细
