# 科室领导 B1 评分工作台设计

**目标**

为 `section-leader` 落地一套独立于员工端的评分工作台。入口按“人”组织，再切到该员工本季度的“目标”，最后在“关键结果”层级逐条评分。评分动作不再依赖单独的“提交本季度评分”按钮，KR 分数保存后立即生效，后续允许继续修改。

**范围**

- 本次只落地科室领导/小组负责人工作台。
- 员工端现有目标维护、KR 维护、材料上传流程保持不变。
- 现有部门领导页面不重做，只保留原有能力。
- 旧的目标维度三段式评分接口保留兼容，但科室领导默认不再使用。

## 交互结构

### 1. 页面入口

- 当前登录角色为 `section-leader` 时，首页主区域切换为负责人工作台。
- 顶部继续保留账号切换、搜索、季度筛选和状态筛选。
- 页面标题从“我的 OKR”调整为“评分工作台”。

### 2. 页面布局

- 顶部是轻量摘要卡，展示：
  - 待评分员工数
  - 待评分关键结果数
  - 已完成评分员工数
  - 当前季度总分概览
- 主体区域采用 `左侧人员列表 + 右侧评分工作区`。

### 3. 左侧人员列表

- 每个员工卡片展示：
  - 员工姓名
  - 科室
  - 所属评价组
  - 本季度目标数 / 关键结果数
  - 已评分关键结果数
  - 当前状态标签
- 状态标签口径：
  - 存在待评分 KR：`待评分`
  - 所有可评分 KR 都已有分数：`已完成`
  - 当前筛选后没有可评分目标：`待补充`

### 4. 右侧评分工作区

- 顶部展示当前员工摘要：
  - 员工姓名、科室、评价组
  - 本季度目标数、关键结果数
  - 季度总分
  - 最近评分时间
- 目标切换区用横向标签页展示当前员工本季度所有目标。
- 选中目标后，下方按 KR 卡片列表展示。

### 5. KR 卡片

每张 KR 卡片展示：

- `KR 编号 + 名称`
- KR 分值
- 完成状态
- 材料数量
- 证明材料列表入口
- 评分输入框
- 负责人备注输入框

评分输入框规则：

- 分值范围为 `0 ~ KR 分值`
- 留空代表暂未评分
- 修改后立即保存
- 后续允许再次修改

备注规则：

- 可为空
- 与 KR 分数一起保存
- 不展示“自动保存中”提示文案，只保留静默保存能力

## 数据规则

### 1. KR 评分

- 新增 KR 级评分写接口。
- KR 评分字段保存到 KR 自身，至少包括：
  - `score`
  - `reviewComment`
  - `reviewerId`
  - `reviewedAt`

### 2. 目标汇总

- 目标总分由该目标下所有 KR 的 `score` 求和得到。
- 目标一旦至少有一条 KR 被评分，即记录 `reviewerId`。
- 目标下所有 KR 都有分数时，目标状态自动变为 `reviewed`。
- 若已评分目标中有 KR 被清空分数，则目标状态回退为 `pending_review`。

### 3. 评分权限

- 科室领导仅可给本部门、本科室员工的 KR 打分。
- 员工仍不能修改锁定目标的 O/KR 内容。
- 员工在 `pending_review` 或 `reviewed` 状态下，仍允许继续补充 KR 材料。
- 科室领导评分不影响员工上传材料能力。

## 兼容策略

- 已存在的老数据若使用旧的目标级评分，仍保留原始 `reviewScore`。
- 新评分工作台只对 `reviewMode = "kr"` 的目标按 KR 分数汇总。
- 未进入 KR 评分的新目标默认采用 `reviewMode = "kr"`。

## 实现方式

- 后端在 `server.ps1` 增加 KR 评分接口与目标汇总同步逻辑。
- 前端增加独立的 `leader-role-overrides.js/.css`，通过覆盖渲染函数的方式为 `section-leader` 输出 B1 工作台。
- 不直接大面积重写员工端页面，避免回归现有交互。
