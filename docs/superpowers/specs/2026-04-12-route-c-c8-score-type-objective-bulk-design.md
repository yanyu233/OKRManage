# Route C C8 评分类型与客观项批量评分设计

## 目标

在 Route C 新架构里补齐一套更贴合真实业务的评分机制：

- 为每条 KR 引入评分类型：`客观评分项` / `主观评分项`
- 员工自建目标时，KR 默认是 `客观评分项`
- 员工将自建 KR 改成 `主观评分项` 时，前端弹确认提醒
- 模板目标支持按 KR 配置评分类型
- 负责人批量评分从“万能全选器”改成“面向客观评分项的批量处理”

## 背景问题

当前 Route C 的批量评分仍然偏“全选后统一赋分”，更像通用工具，不够贴合业务实际。真实场景里：

- 员工自建目标多数是客观结果，通常适合批量处理
- 模板目标中的部分 KR 是主观评分项，需要逐条人工评分
- 负责人更关心“哪些客观项可以快速批量处理”，而不是手工圈选所有层级对象

所以判断一条 KR 是否适合批量评分，不应该看目标名称是否来自模板，而应该看 KR 本身的评分类型。

## 方案

### 1. 数据模型

新增统一评分类型字段：

- `GoalTemplateKeyResult.scoreType`
- `KeyResult.scoreType`

取值固定为：

- `objective`
- `subjective`

默认值：

- 模板 KR：默认 `subjective`，由系统管理员显式配置
- 员工自建 KR：默认 `objective`
- 模板导入生成的员工 KR：继承模板 KR 的 `scoreType`

### 2. 系统管理员端

模板目标配置页中，每条模板 KR 增加“评分类型”下拉：

- 客观评分项
- 主观评分项

保存规则：

- 沿用现有模板保存入口
- 同时持久化 `scoreType`
- bootstrap 也要把 `scoreType` 回传给前端

### 3. 员工端

补一个最小可用的新建目标入口，放在“我的 OKR”页。

能力范围：

- 仅支持新建季度目标
- 支持填写目标名称、说明
- 支持增删 KR
- 支持设置 KR 名称、说明、分值、评分类型

规则：

- 新增 KR 默认 `客观评分项`
- 如果员工把某条 KR 从 `客观评分项` 改为 `主观评分项`，弹出确认框，提示语固定为：
  - `员工自定目标一般为客观评分项，由绩效小组核实是否完成后赋分`
- 用户确认后才允许切换
- 新建目标保存为 `draft`
- 新建目标在本季度目标序列中排到模板目标之后，继续沿用现有 `O1/O2/O3...` 编号重排逻辑

### 4. 负责人工作台

批量评分改成“客观项批量评分”，规则如下：

- 批量评分入口仍在评分工作台
- 批量评分只会命中 `scoreType=objective` 的 KR
- 如果用户手动选中的 KR 里包含 `subjective` 项，这些项自动跳过
- 跳过原因新增 `subjective-only`

交互保持现有筛选方式，但文案和行为明确为“客观项批量评分”：

- 科室筛选
- 小组筛选
- 员工筛选
- 全选员工
- 全选目标
- 全选关键结果
- 全选（去除模板目标）

语义解释：

- 全选目标 / 全选关键结果，本质上只会选中当前范围内的客观项 KR
- 模板目标整体仍可通过“去除模板目标”快速排除
- 模板目标中的客观项 KR 在不排除模板目标时仍可以进入批量池
- 主观项始终留在逐条评分流中

### 5. 详情展示

员工端和负责人端的 KR 列表都显示评分类型标签，便于快速识别：

- 客观评分项
- 主观评分项

## 验证重点

### 后端

- 模板 KR 的 `scoreType` 可保存并正确回显
- 模板导入后，员工 KR 继承模板 `scoreType`
- 员工新建目标时，未显式指定的 KR 默认 `objective`
- 负责人批量评分只更新 `objective` 项
- `subjective` 项被跳过时，返回 `subjective-only`

### 前端

- 模板目标配置页可编辑评分类型
- 员工新建 KR 默认显示“客观评分项”
- 员工改成“主观评分项”时弹确认提醒
- 负责人批量评分弹窗文案改成客观项导向
- 负责人查看页面能区分客观项 / 主观项
